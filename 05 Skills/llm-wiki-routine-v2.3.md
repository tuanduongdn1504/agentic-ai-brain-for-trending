## §1. Library-vocab observational item layer (formalized)

### What it is

A **Library-vocab observational item** is a methodology-vocabulary observation that:

- Captures a recurring frame, discipline, or measurement-axis (not a structural property)
- Is corpus-recurring at N≥1
- May or may not become a top-level Pattern; some remain useful at N=2-3 even within single vertical
- Has explicit promotion-eligibility windows + retire-checks
- Lives at the **OC (observational candidate) layer** until promoted

### When to use Library-vocab vs Pattern sub-mechanism vs sub-archetype

| Vehicle | Use when |
|---|---|
| **NEW top-level Pattern** | Structural recurring property at N≥3 cross-tier cross-vertical |
| **Existing Pattern sub-mechanism** | New mechanism-axis within established Pattern's scope; N≥2 within Pattern |
| **NEW sub-archetype** | Structural variant within existing tier's archetype layer; N≥1 with clear typological distinction |
| **NEW sub-sub-mechanism** | Mechanism-layer-distinction within existing sub-mechanism; N≥1 with clear measurement-axis distinction |
| **Library-vocab observational item** | Methodology/discipline/measurement-axis observation; N≥1; structural-membership not load-bearing; vocabulary item useful even at N=1 |

- v87 "Token-Economy-Quantification per Communication Style" = Library-vocab (methodology-axis, not structural)

## §2. Library-vocab promotion-eligibility ladder

### Threshold ladder

| Stage | N | Status | Audit action |
|---|---|---|---|
| Registration | 1 | PROVISIONAL N=1 | First-cycle stale-check at next audit |
| First N=2 strengthening | 2 | PROVISIONAL N=2 | Watch for N=3 promotion-eligibility |
| Promotion-eligibility | 3 | **N=3 PROMOTION-ELIGIBLE** | Evaluate at audit for CONFIRMED |
| Stronger eligibility | 4 | **N=4 STRONGER PROMOTION-ELIGIBLE** | Default PROMOTE at audit |
| Confirmed | ≥3 or ≥4 (depending on cross-axis spread) | **CONFIRMED Library-vocab** | Standalone vocabulary item |

### Cross-axis spread requirements at promotion

For PROMOTION from PROMOTION-ELIGIBLE → CONFIRMED:
- **N=3 cross-author**: at least 3 distinct authors
- **N=3 cross-vertical**: at least 3 distinct verticals (tier-archetypes)
- **N=3 cross-scale (optional but preferred)**: range of subject scales (micro/mid/macro)

If N=3 within single vertical only → hold at PROMOTION-ELIGIBLE; needs cross-vertical for CONFIRMED.

├─ sub-archetype (variant within tier)

│  │  ├─ sub-sub-mechanism (mechanism-layer-distinction within sub-mechanism)

## §5. Operator-Elected Phase 0.9 OVERRIDE policy (Library-vocab #15 codified)

### What it is

A protocol enabling the operator to **explicitly OVERRIDE Phase 0.9 STRICT verdicts** in cases where the routine v2.2 (or v2.3) filter would reject a wiki subject but the operator judges the corpus-value worth the routine-discipline cost.

### When NOT to use

- For convenience or impatience — OVERRIDE is NOT a "skip the filter" shortcut
- Without explicit honest-documentation-trail — operator must record the override + reasoning
- For subjects that fail multiple criteria at WEAK level with no compensating signal

### When to use (per v84 precedent)

- The subject has STRONG corpus-knowledge value AND meets only 1 PASS criterion + 1 PROBABLE-PASS criterion
- The override is declared **before** wiki build begins (not post-hoc)
- Honest-documentation-trail maintained throughout all wiki artifacts:
  - CLAUDE.md project context flagged as OVERRIDE
  - 02 Wiki/index notes OVERRIDE-INCLUDE status
  - State entries carry OVERRIDE-INCLUDE-EVIDENCE-QUALITY-CAVEAT
  - Streak notation reflects the override (Option B2)

### Documentation discipline (load-bearing)

Every wiki built under OVERRIDE must explicitly:

1. **State the verdict** clearly: "Phase 0.9 STRICT verdict = X PASS criteria; OPERATOR ELECTED OVERRIDE"
2. **State the override reasoning**: what corpus-value justified the override
3. **State the one-time-or-precedent declaration**: is this a one-time-exception (default) or precedent-setting?
4. **Flag all derived registrations** with OVERRIDE-INCLUDE-EVIDENCE-QUALITY-CAVEAT for future retire-checks
5. **Continue streak with B2 notation** (see §6)

### v84 anchor case

v84 image-blaster: 0 clear PASS + 1 WEAK + 1 PROBABLE-PASS-on-revision. Operator declared one-time-exception. NEW T1 sub-archetype "Linear-Workflow-Pipeline Skill Bundle" registered with EVIDENCE-QUALITY-CAVEAT. v85 + v87 + v88 + v89 all clean PASS = discipline-restoration durable at 4-consecutive (CORPUS-RECORD).

### Operator-Elected-Override-Policy as Library-vocab #15 CONFIRMED N=1 ONE-TIME-EXCEPTION

## §6. Streak notation Option B2 ("N+1*" with override-asterisk)

### Context

Pre-v84: Phase 0.9 streak was tracked as simple integer count of consecutive PASS verdicts (e.g. "19 consecutive PASS").

v84 OVERRIDE created a streak-treatment decision: how do you count a forced-INCLUDE?

### Three options considered at v85 audit

| Option | Treatment | Tradeoff |
|---|---|---|
| **B1 RESET** | Streak resets to 0 at OVERRIDE; v85 starts new 1-consecutive | Simple; punishes operator for using OVERRIDE |
| **B2 Streak-with-override-asterisk** | Notation "N+1*" where N = clean PASS count and 1* = OVERRIDE marker | Honest; preserves momentum-tracking + override-frequency tracking |
| **B3 Verdict-launder** | OVERRIDE counts as PASS for streak; no asterisk | Dishonest; obscures override-frequency |

### Operator decision at v85: **Option B2** ADOPTED

Rationale:
- **Honest** — override-frequency-tracking preserves the v2.3-trigger condition (§7)
- **Momentum-preserving** — operator isn't punished for using OVERRIDE responsibly
- **Audit-friendly** — clearly distinguishes clean-PASS streak from override-included streak

### Notation examples

- "19+1*" at v85 = 19 clean PASS (v65-v83) + 1 OVERRIDE (v84) + 1 clean PASS (v85) = 20 PASS + 1 OVERRIDE
- "20+1*" at v85 ship (post-v85 = 20 clean PASS in v65-v85 window + 1 OVERRIDE)
- "23+1*" at v89 ship = 23 clean PASS + 1 OVERRIDE

### Override-frequency calculation

In an N-instance window, override-frequency = OVERRIDEs / N. Trigger thresholds in §7.

## §8. PROBABLE-PASS criterion call discipline

### What it is

A **PROBABLE-PASS** verdict on a Phase 0.9 (a) criterion is used when the evidence is **inference-tolerant** (strong signals but not direct profile-field confirmation).

### When to use

- VN-native-fluency evidence in idiomatic phrasing → PROBABLE-PASS (a) for inferred-VN-located
- Chinese-handle-inferred + code-switching → PROBABLE-PASS (a) for Chinese-Mainland-inferred
- Community-org self-positioning + member-fluency → PROBABLE-PASS (a) for cultural-cluster-membership
- Product-locale-inclusion at first-class i18n → PROBABLE-PASS (a) for locale-inclusion-as-cluster-membership

### When NOT to use

- Single weak signal alone → use WEAK PASS or FAIL
- Speculation without anchor evidence → FAIL

### Recording discipline

PROBABLE-PASS is recorded as **PASS-equivalent** for the criterion count but **flagged for v95 verification** at next audit. If new evidence at v95 confirms or refutes the inference, the verdict is re-anchored.

### Worked examples

- v73 cc-switch farion1231: Chinese-handle inference + code-switching = PROBABLE-PASS (a) ✓
- v77 datawhalechina: Community-org self-positioning + Tsinghua-academic-anchor = PROBABLE-PASS (a) ✓
- v78 ECC: `docs/vi-VN/README.md` locale-inclusion = PASS (a) via locale-inclusion sub-criterion

## §19. Pattern #19 CONSOLIDATION discipline (graveyard-relief)

### Trigger condition

When a Pattern sub-mechanism layer reaches **10+ candidates with 9+ at PROVISIONAL N=1** ("graveyard"), the next audit MUST evaluate cross-candidate CONSOLIDATION.

### Consolidation methodology

1. **Identify cross-candidate themes**: cluster N=1 candidates by shared structural property (locale, profile-type, technical-domain, etc.)
2. **Propose unified sub-mechanism**: if a cluster reaches N≥3 cross-candidate, propose unified sub-mechanism with the original candidates as sub-variants
3. **Promote unified sub-mechanism to CONFIRMED at N=cluster-size**
4. **Retire absorbed candidates** (the N=1 originals are now sub-variants of the unified mechanism)

### v90 worked example: Pattern #19 CONSOLIDATION

**Pre-audit**: Pattern #19 sub-mechanism layer = 12 candidates with 11 at PROVISIONAL N=1.

**Cluster identified**: 5 VN-related candidates (19j Diaspora + 19k Located-Solo + 19n Indie-Org + v87 Solo-Multi-Product + v88 Multi-Org-Founder).

**Unified sub-mechanism**: "VN-Community Multi-Profile-Type" CONFIRMED N=5 with 5 profile-type sub-variants (19-VN-a through 19-VN-e).

**Result**: 12-candidate-layer / 11-N=1 → 7-candidate-layer / 6-N=1 + 1 CONFIRMED N=5 sub-mechanism.

### When NOT to consolidate

- If the 10+ candidates have no shared structural property (truly distinct sub-mechanisms)
- If consolidation would obscure load-bearing distinctions
- If the cluster is <3 candidates
