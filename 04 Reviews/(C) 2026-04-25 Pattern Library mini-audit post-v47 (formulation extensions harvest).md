### Extension 2: Pattern #28 Multi-Provider AI Support — verification-not-routing 3rd sub-axis NEW

**Pre-audit status:** CONFIRMED (promoted v25 audit). 2 implementation variants documented in formal statement:

1. **Abstraction-library variant** — adopt libraries like LiteLLM / OneAPI / OpenRouter / LangChain ChatModels (e.g., TrendRadar v19 LiteLLM 100+ providers; crawl4ai v29 `unclecode-litellm` fork; ruflo v42 cost-based routing)
2. **Native BYO variant** — framework directly supports multiple providers configurable per-request (e.g., multica v15 / Skyvern v24 / OpenHands v30 / DeepTutor v38 / browser-use v41 ~115-provider surface)

11 data points cumulative at pre-audit (v19-v47). Both variants understood as solving the same vendor-decoupling problem at different abstraction layers (library vs framework).

**v47 aidevops observation:** **Cross-provider verification-not-routing** — destructive operations (force push, production deploy, data migration) are verified by a second cross-provider model before execution. Verbatim from aidevops repo: *"different providers have different failure modes, so correlated hallucinations are rare."* 4-provider OAuth pool (Anthropic + OpenAI + Cursor + Google) cycles for routing; verification specifically invokes second model from **different provider** for destructive-operation safety check.

This is **structurally distinct** from prior #28 sub-axes:

- **Routing/abstraction (28-routing):** SECOND provider invoked when first is unavailable (failover) OR cheaper-with-acceptable-quality (cost optimization) OR has-needed-capability (feature routing). Primary-secondary distinction is fluid; both producing inference output for end-user-visible task.
- **Native multi-provider (28-native):** SAME architectural role; just multiple SDKs available. User configures which to use per-task.
- **Verification (28-verification NEW):** SECOND provider invoked specifically as **independent cross-validation** before destructive action. Primary provider produces proposed action; secondary provider (from **different vendor**) reviews proposed action for hallucination/error before execution. Vendor-difference is critical (different training data + different failure modes = uncorrelated hallucinations).

**Proposed Pattern #28 3-axis taxonomy:**

| Sub-axis | Mechanism | Primary motivation | Vendor-diversity requirement | Corpus evidence |
|----------|-----------|---------------------|------------------------------|-----------------|
| **28-routing** | Abstraction library or native multi-provider for runtime routing | Cost / availability / feature diversity | Helpful, not strictly required | TrendRadar v19 LiteLLM, crawl4ai v29 fork, ruflo v42 cost-routing |
| **28-native** | Direct integration with multiple provider SDKs | User flexibility / no abstraction overhead | Helpful, not required | multica v15, Skyvern v24, OpenHands v30, DeepTutor v38, browser-use v41 |
| **28-verification NEW** | Cross-provider invocation as independent safety check before destructive operations | Hallucination-resilience for high-stakes operations | **Required** — different vendors needed for uncorrelated failure modes | **aidevops v47 (corpus-first; N=1)** |
