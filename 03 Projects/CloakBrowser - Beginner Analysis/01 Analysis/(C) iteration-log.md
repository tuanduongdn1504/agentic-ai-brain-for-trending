## 3. Surprises + corrections

### Surprise 1: README claims "49 patches" while CHANGELOG shows "57 patches"

Initial assumption: README would be the authoritative source for current patch count. CHANGELOG v0.3.25+ documents "57 patches" but README header still says "49 source-level C++ patches."

This is a **synchronicity-discipline gap** — README and CHANGELOG are out of sync. Worth noting as minor disclosure-honesty observation but not pattern-changing. Possibly:
- v68 zero's "Synchronicity-Discipline-As-Policy" OC-E counter-evidence (CloakBrowser does NOT exhibit synchronicity discipline as v68 zero does)
- Or simply incremental-update lag (frequent releases mean README updates lag CHANGELOG)

**Lesson captured:** When checking subject self-positioning claims, cross-reference README vs CHANGELOG. Discrepancies are minor observation evidence about project maturity / discipline.

### Surprise 2: $100 max aggregate liability cap is corpus-rare

Initial expectation: Standard OSS license shapes use unbounded "AS IS, NO LIABILITY" framing. Most prior corpus Pattern #45 subjects (Unsloth + AutoGPT) follow this.

CloakBrowser's bounded liability cap ($100 max aggregate) is **corpus-rare**. More typical of B2B SaaS contracts than OSS license files. Implications:
- Defensive against statutory damages in some jurisdictions
- Signals B2B revenue model where actual liability commitments require separate licensing
- Acknowledges nonzero baseline risk that pure "AS IS" framing doesn't capture

**Lesson captured:** Liability framing is a structural license-axis dimension worth tracking. Add to v2.3 routine codification candidates: "liability-framing axis" in Phase 0 13-axis classification (currently subsumed under "License" axis but warrants separate sub-tracking).

### Surprise 3: 3-tier signature verification (GPG + GitHub attestation + cosign) corpus-broadest

Initial expectation: Most corpus subjects use 0-1 verification mechanisms (typical: just package-registry checksums).

CloakBrowser documents **3-tier signature verification** explicitly in README:
1. GPG signature on git tags
2. GitHub binary attestation (`gh attestation verify`)
3. Cosign Docker image signature

This is **corpus-broadest signature-verification surface** documented in a single subject. Pattern #66 meta-supply-chain-awareness gets significant strengthening evidence — potential sub-archetype "signature-verification-as-marketed-feature."

**Lesson captured:** Subjects that prioritize supply-chain trust (dual-use tools, stealth tools, security-adjacent tools) document verification more explicitly. Pattern #66 sub-mechanism candidate.

### Correction 1: Initial classification considered T1 sub-archetype #7

**Initial deliberation:** Brief consideration of whether CloakBrowser could be classified within v68's NEW T1 sub-archetype #7 "programming-language-as-agent-substrate" since CloakBrowser is a foundational substrate (browser binary) for agent workflows.

**Correction:** No. T1 sub-archetype #7 has 4 distinguishing properties (primary-artifact-is-language, agents-as-primary-users, language-level-agent-readability-commitments, skill-content-bundled-with-runtime). CloakBrowser fails at least 3:
- Primary artifact is browser binary, not programming language (FAIL)
- Agents are users but NOT the primary design assumption (web automation generally is the primary use) (FAIL)
- No language-level agent-readability commitments (FAIL)
- No skill-content-bundled-with-runtime mechanism (FAIL — CloakBrowser doesn't have a skills protocol presence at all)

**Correction outcome:** Classify CloakBrowser as **T2 Service** (definitive, no provisional needed). Pattern #86 T1 sub-archetype #7 remains N=1 at v68 zero subject only.

**Initial deliberation:** Considered whether CloakHQ's 6-repo org (3 originals + 3 forks) qualifies as Pattern #19 ecosystem-portfolio-builder N+1 evidence.
