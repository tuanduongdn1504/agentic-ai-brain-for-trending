## 3. Surprises + corrections

### Surprise 1: 91 releases in 5 months at solo scale

The release cadence (~1.75 days) is corpus-high for non-corporate solo authors. CHANGELOG narrative reveals this is sustainable because:
- Each release has narrow scope (1-5 Fixed entries + small Added feature)
- Author-written CHANGELOG (no automation)
- No external release-coordination overhead
- Continuous defense-against-vendor-enforcement workload generates frequent small fixes

This is **not** "ship more often" advice but evidence of viability conditions. Captured as Decision-4-style observation in v2.2 codification queue for v2.3.

### Surprise 2: Corpus-record fork_ratio (1/10,500)

Initially assumed the 1 fork was a WebFetch error. Re-probed with verification-focused prompt: confirmed 1 fork. This is 1,780× lower than prior corpus-record (v65 = 0.178). Captured as Library-vocabulary item #11 candidate.

Mechanism hypotheses (3 competing):
1. **Drive-by-stars dominant** — plugin functionality + TOS-violation appeal drives star-as-bookmark without clone-to-fork intent
2. **Tool-anti-fork architecture** — plugin design (single-file npm install) discourages forking; users install via `opencode-antigravity-auth@latest` not fork-and-customize
3. **Recency** — virality may be recent (within last 1-2 months); forks lag

All three plausible. Library-vocabulary observation flags the metric without committing to mechanism.

### Surprise 3: 5 disclosure surfaces in single subject

When cataloging Pattern #83 evidence, expected 1-2 disclosure surfaces per registration norm. Found 5 (README `[!CAUTION]` + README Legal footer + Intended Use + docs/ANTIGRAVITY_API_SPEC.md status banner + CHANGELOG entries). The within-subject saturation is itself a corpus-first observation — Pattern #83 can be "fully satisfied" by a single subject's saturation rather than requiring evidence accumulation across multiple subjects.

Captured as "Multi-disclosure-surface saturation as N=3 evidence shortcut" routine v2.3 candidate.

### Correction 1: Initial Pattern #85 N+1 evaluation

**Initial assumption:** Maybe v67 provides N+1 evidence for Pattern #85 Platform-Primitive Foundation (since plugin uses `@opencode-ai/plugin` heavily).

**Correction:** No. Pattern #85 requires (a) exclusive infrastructure delegation, (b) self-as-instance framing, (c) host-unit self-description, (d) host-CLI extensibility. opencode-antigravity-auth uses `@opencode-ai/plugin` as a dependency but does NOT self-identify as a running instance of Opencode, does NOT count internals in Opencode units, and does NOT extend via Opencode CLI. Ordinary heavy-dependency plugin, not Platform-Primitive Foundation.

**Lesson:** Plugin-using-platform-SDK ≠ Platform-Primitive-Foundation. The diagnostic is the SELF-AS-INSTANCE framing + host-unit self-description, not the SDK dependency. Pattern #85 stale-check at v68 retains its N=1 status (does not stale-via-counter-evidence — v67 simply doesn't qualify).
