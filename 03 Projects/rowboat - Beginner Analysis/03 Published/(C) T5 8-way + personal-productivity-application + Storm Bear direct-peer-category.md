## Part 2: Personal-productivity-application sub-archetype formalization

### 2.1 Formal definition

**Personal-productivity-application (NEW T5 sub-archetype at v43):**

> An application deployed to end-user device/OS, operates on user's personal data (emails, calendar, notes, meeting transcripts), maintains a persistent knowledge artifact (markdown vault / knowledge graph / personal wiki), augments user's own work (not substitutes autonomous agent), with local-first data stewardship discipline.

### 2.2 Distinguishing criteria (7 tests)

| Test | Personal-productivity-application | Counter-example (other T5) |
|---|---|---|
| Operates on user's own personal data? | Yes (Gmail + Calendar + Meeting notes) | Browser-automation operates on external websites |
| Maintains persistent artifact for user? | Yes (local Markdown vault) | SWE-agent modifies arbitrary codebases |
| Augments user's work (not autonomous)? | Yes (AI coworker framing) | paperclip "zero-human companies" |
| Local-first data stewardship? | Yes (explicit in README) | Cloud-first (most commercial T5) |
| Deployment target = end-user device? | Yes (Mac/Windows/Linux binaries) | Docker/cloud (most T5) |
| User is primary operator (not dev)? | Yes (consumer-grade UI) | SDK/CLI (most T5 commercial tools) |

**Unique Storm Bear capability:** Pattern Library (cross-wiki synthesis layer). Rowboat does NOT produce pattern-level observations — it does entity extraction + note generation. Storm Bear's pattern-library function is a **human-meta-cognitive layer** that rowboat's AI Copilot does not replicate.

**Unique rowboat capability:** Continuous knowledge-graph maintenance (24/7 compounding). Storm Bear's knowledge compounds only during operator sessions.

### 3.2 Empirical test thesis

**Storm Bear operational hypothesis:**
> *"instead of re-deriving knowledge from scratch on every query, the LLM incrementally builds and maintains a structured wiki — summaries, entity pages, cross-references, evolving synthesis. I curate sources, direct analysis, and ask the right questions. The LLM handles all the grunt work."*
> (Storm Bear vault CLAUDE.md)

**Rowboat operational hypothesis:**
> *"context accumulates over time, relationships are explicit and inspectable, notes are editable by you, not hidden inside a model, everything lives on your machine as plain Markdown. The result is memory that compounds, rather than retrieval that starts cold every time."*
> (Rowboat README verbatim)

**Empirical question:** Which approach compounds knowledge faster + at higher quality?

**Pre-pilot hypothesis (Storm Bear's bet):**
- Human curation produces higher-quality cross-references, more-intentional entity pages, fewer hallucinations
- LLM excels at pattern-recognition across wikis (Pattern Library function) once humans curate entity level

**Pre-pilot hypothesis (Rowboat Labs' implicit bet):**
- AI coworker can maintain knowledge graph at continuous pace humans cannot match
- Quality control emerges through user-oversight + iteration (not up-front curation)

**Resolution protocol:** 4-week pilot (see Storm Bear pilot relevance section below). Result:
- Storm Bear correct → pattern library discipline vindicated; rowboat observational for continuous-sync feature only
- Rowboat correct → Storm Bear hybrid-adopts AI maintenance
- Both correct for different layers → Storm Bear continues patterns; rowboat handles sync
