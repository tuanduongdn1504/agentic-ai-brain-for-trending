## 4. Vibe-design philosophy

### Aesthetic-first descriptive language

README descriptive language (per WebFetch):
- *"warm terracotta"*
- *"void-black canvas"*
- *"cinematic dark UI"*

These are **vibes not specs** — aesthetic intent captured via metaphor, not deterministic parameters (RGB values, exact font sizes).

### DESIGN.md as vibe-specification

DESIGN.md balances specification (color tokens, typography) with vibe (agent prompts section captures aesthetic intent). Hybrid: **spec + vibe**.

### Embrace, not reject, of vibe-coding

Unlike spec-kit v17 which explicitly anti-"vibe coding," awesome-design-md explicitly embraces aesthetic-first approach.

## 5. Philosophical contrast — spec-kit v17 vs awesome-design-md v25

### spec-kit v17 position (anti-vibe)

> *"focus on product scenarios and predictable outcomes instead of vibe coding every piece from scratch"*

- Constitution with 9 articles
- Compile-time gates
- SDD (Spec-Driven Development) methodology
- Predictability > creativity
- Deterministic spec → deterministic output

### awesome-design-md v25 position (pro-vibe)

- Aesthetic-first descriptive language
- "Vibe-coding" + "vibe-design" as explicit topic tags
- Agent prompts capture aesthetic intent
- Creative interpretation > rigid reproduction
- Vibe → pixel-perfect match (LLM bridges the gap)

### Corpus-first explicit pair

**First vibe-pro vs vibe-anti pair in Storm Bear corpus.** Prior corpus had implicit spectrum (constitutional paperclip v14 + BMAD v11 methodology vs more flexible tools) but never explicit pair on the vibe axis.
