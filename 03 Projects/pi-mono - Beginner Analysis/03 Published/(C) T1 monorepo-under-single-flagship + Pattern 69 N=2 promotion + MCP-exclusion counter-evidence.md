## Part 1: pi-mono as T1 monorepo-under-single-flagship — a new solo-T1 sub-archetype

### 1.1 The T1 scope-axis at N=13

Post-claude-hud v35, T1 had 12 entrants. At v35, a new scope-axis sub-classification was introduced: **broad-methodology ← mid-skill-library → narrow-display-primitive**. claude-hud occupied the narrow end; prior entrants occupied broad-methodology (spec-kit v17 / BMAD v11 / GSD v5) or mid-skill-library (agency-agents v18 144-agent library / codymaster v12 79-skill library / OMC v27 19-agent multi-runtime).

**pi-mono (v36) doesn't fit cleanly on this axis.** It is:
- Not narrow (38.9K stars + 7 packages + 110 user-facing primitives + 20+ providers)
- Not a methodology framework (anti-methodology positioning: *"Adapt pi to your workflows, not the way around."*)
- Not a skill-library (pi-skills is a sibling repo, not a pi-mono-internal skill library)
- Not a tutorial (claude-howto v32 / Everything Claude Code v1)

**pi-mono is instead: a solo-author open-source coding-agent engine + ecosystem + flagship CLI + session-data-donation flywheel bundled into a single monorepo.** This is structurally different from all prior T1 entrants.

### 1.2 Proposed new sub-archetype label

**"T1 monorepo-under-single-flagship"** — working label.

Criteria (from pi-mono observation):
1. **Multi-package publication** — 3+ npm packages (pi-mono has 7) under single author's npm scope
2. **Single flagship** — README directs to one primary package (pi-coding-agent) as user entry-point
3. **Foundation-layer + spin-off structure** — packages fall into 2+ categories (foundation vs product)
4. **Lockstep versioning** — all packages share version number + release together
5. **Solo maintainer** — no corporate/LLC backing; single GitHub user is majority committer

pi-mono meets all 5 criteria.

## Part 2: Pattern #69 Agent-PR Fast-Track Governance Protocol — N=2 promotion analysis

### 2.1 Candidate #69 registration (v31 awesome-mcp-servers)

Pattern #69 was registered at v31 as candidate with N=1 (stale-risk-flagged at registration per v2.1 discipline). Formal statement at registration:

> **#69 Agent-PR Fast-Track Governance Protocol** — OSS projects introducing explicit opt-in protocols for agent-authored contributions (e.g., `🤖🤖🤖` PR-title suffix allowing agent-authored PRs to bypass standard review discipline).

**v31 data point:** awesome-mcp-servers CONTRIBUTING.md documents `🤖🤖🤖` suffix opt-in.

**Stale-risk threshold:** +5 wikis from registration (v36) for stale-flag review; +10 (v41) for retirement.

### 2.2 v36 pi-mono observation — `lgtm` / `lgtmi` maintainer-approval keywords

From AGENTS.md §Contribution Gate (verbatim):

> - `lgtmi` approves future issues
> - `lgtm` approves future issues and rights to submit PRs

Semantics:
- **`lgtmi`** ("looks good to me — issues") — exempts commenter from future issue auto-closure
- **`lgtm`** ("looks good to me") — exempts from issue auto-closure AND grants PR-submission rights

Enforced by GitHub Actions workflow `approve-contributor.yml` which parses these comment keywords posted by maintainers on new-contributor issues.

### 2.3 Structural comparison: 69a vs 69b

| Dimension | 69a `🤖🤖🤖` (v31 awesome-mcp-servers) | 69b `lgtm`/`lgtmi` (v36 pi-mono) |
|-----------|----------------------------------------|----------------------------------|
| **Author side** | Contributor-authored opt-in (contributor decides) | Maintainer-authored approval (maintainer decides) |
| **Trigger moment** | At PR creation (title suffix) | At comment time on issue or PR (keyword) |
| **Granularity** | Per-PR (suffix on each) | Persistent per-contributor (one keyword = ongoing rights) |
| **Enforcement** | Project-side fast-track (less review) | Project-side unlock (standard review still applies) |
| **Purpose** | Separate agent-authored vs human-authored volume | Gate new-contributor inbound to reduce maintainer-load |
| **Problem framing** | "Too many agent PRs; fast-track the obvious ones" | "Too many low-quality new contributors; pre-filter by approval" |

**Both are agent-era-OSS-governance-gate mechanisms targeting the LLM-contribution-volume problem**, but from opposite sides:

- **69a** — contributor-side honesty-signal; projects opt into accepting agent-volume with reduced review
- **69b** — maintainer-side vetting-gate; projects preemptively auto-reject new-contributor volume and selectively unlock

## Part 3: Pattern #18 Agent Runtime Standardization — pi-mono MCP-exclusion counter-evidence at T1 scale

### 3.1 Pattern #18 current state

Pattern #18 (CONFIRMED v15, strengthened v17 to "community-platform-specific" with corporate-official opt-out refinement, strengthened v31 mini-audit to triple-layer formulation):

> **#18 Agent Runtime Standardization** — Three-layer formulation:
> - **Layer 1:** MCP (Model Context Protocol, Anthropic-published) = universal tool/resource-sharing standard adopted across coding-agent ecosystem
> - **Layer 2:** OpenClaw + Hermes = community-platform-scoped agent-runtime standards (9+ MCP-consuming corpus data points post-v31)
> - **Layer 3:** Per-runtime-specific (Claude Code `/plugin marketplace` / Cursor config / etc.)

### 3.2 pi-mono counter-evidence at T1 scale

**First observation:** pi-coding-agent at 38.9K T1 stars deliberately excludes MCP. From coding-agent docs:

> Build CLI tools with READMEs (see Skills), or build an extension that adds MCP support.

This is **prescriptive, not descriptive** — pi is not MCP-less because it hasn't gotten to it. Mario designed pi to route around MCP by:

1. **CLI tools with READMEs** — model reads tool's README via bash, calls tool via bash. No standardized protocol; plain command invocation.
2. **Agent Skills standard** — declarative, hot-reloadable, cross-runtime-compatible (pi-skills sibling explicitly Claude Code + Codex CLI compatible)
3. **Extensions (TypeScript)** — if someone wants MCP support in pi, they write an extension that adds it. Not core.

### 3.3 Why this matters for Pattern #18

**Counter-evidence scope:** pi-mono is the **first T1-coding-agent-runtime in corpus that deliberately rejects MCP-as-primary at scale.**

Prior "non-MCP" corpus data points:
- **claude-hud v35** (20K T1) — narrow display-layer plugin; extension-point consumer of Claude Code statusline API, not MCP. But claude-hud is narrow-scope (single output surface); not architecturally rejecting MCP, just out-of-scope for MCP.
- **awesome-mcp-servers v31** (85K outside-scope) — meta-reference TO MCP, not rejection of MCP.
- **agency-agents v18** (82.9K T1) — does not explicitly reject MCP; agent library operates at different abstraction layer.

**pi-mono is the first architectural rejection of MCP at T1-coding-agent scope.** This is a stronger signal than claude-hud's tangential non-use.

### 3.4 Does this warrant Pattern #18 refinement?

**At N=1 of counter-evidence:** No. v2.1 discipline requires N=2 for refinement.

**What to watch for:** If a 2nd T1-coding-agent-runtime (new wiki in corpus) explicitly rejects MCP at scale, formalize Pattern #18 refinement:

> **Layer 1 refinement (proposed, NOT active at v36):** MCP is a dominant-but-not-universal coding-agent-ecosystem standard. Some T1-coding-agent-runtimes explicitly route around MCP via alternative primitive layers (CLI-tools + Agent-Skills + custom extensions). MCP-adoption is a framework-philosophy choice, not an ecosystem-wide inevitability.

**At v36:** observational counter-evidence; logged for future audit.

### 3.5 Why Mario excludes MCP (architectural hypothesis)

Not explicitly stated in pi-mono docs, but inferable from design:

1. **MCP adds coupling** — standard protocol = shared assumptions between runtime + server. Mario's design philosophy (AGENTS.md: minimal defaults + deep extensibility) may prefer decoupling.
2. **CLI-tools-with-READMEs is simpler** — any CLI tool with a sensible `--help` can be used by an LLM that has bash-access. No MCP-server write-up needed.
3. **Extensions are more flexible than MCP** — pi extensions are TypeScript modules with full ExtensionAPI; can do anything MCP can + more (UI interaction, state persistence, command registration). MCP is more narrowly scoped to tool/resource sharing.
4. **MCP server bus factor** — each MCP server is a separate process/repo with its own maintenance. pi's CLI-tools-with-READMEs model offloads that cost to the tool's own OSS maintenance.

**Cost Mario pays:** Users with existing MCP investments face friction. Pi is less immediately adoptable for MCP-heavy workflows.

### 3.6 Implications for Storm Bear

Storm Bear currently doesn't use MCP servers extensively (Claude Code skill-based + file-based workflows). pi's MCP-exclusion is NOT a blocker. But understanding this design decision:
- Validates one alternative path (non-MCP at T1 scale CAN work at 38.9K stars)
- Reduces dependence-on-MCP assumption in future operator decisions
- Informs evaluation if 2nd T1 MCP-exclusion emerges (coordination signal)
